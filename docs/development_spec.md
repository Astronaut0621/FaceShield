# FaceShield 开发说明书

## 1. 项目目标

FaceShield 是一个面向 AI 换脸诈骗场景的伪造人脸检测系统。系统以单张图片级检测为主，用户上传一张 JPG/PNG 待检测图片后，系统完成人脸预处理、PaddlePaddle 模型推理、伪造概率输出、Grad-CAM 可疑区域热力图生成、检测记录保存和历史记录查询。

本阶段核心目标是先完成可演示的 MVP：

```text
图片上传
-> 人脸检测与裁剪
-> 频域-空域融合模型推理
-> real/fake 判断
-> 伪造概率与风险等级展示
-> 可疑区域热力图展示
-> 检测记录入库
-> 历史记录查询
```

MVP 不要求用户同时上传原图和 AI 换脸图进行对比，也不把视频上传检测作为主流程。训练阶段会使用 real/fake 两类样本学习伪造特征；系统使用阶段只需要一张待检测图片。

## 2. 系统模块划分

| 模块 | 负责人方向 | 核心职责 |
|---|---|---|
| 算法模块 | 算法开发 | 数据集处理、人脸裁剪、FFT 特征、PaddlePaddle 模型训练、评估、推理、Grad-CAM |
| 后端模块 | 后端开发 | 登录认证、FastAPI 接口、文件上传、模型调用、结果封装、记录入库、历史查询 |
| 前端模块 | 前端开发 | 登录页面、图片上传页面、检测结果页面、热力图展示、历史记录页面 |
| 数据库模块 | 数据管理 | 用户表、MySQL 表设计、检测记录留存、模型版本管理、数据查询 |
| 测试与文档 | 全体成员 | 功能测试、接口测试、模型实验记录、说明文档、答辩材料 |

## 3. 算法模块

### 3.1 数据准备

算法模块使用 FaceForensics++ c23 视频子集作为数据来源。当前优先使用：

- `original_sequences/youtube/c23/videos`：真实人脸视频，标签为 `real / 0`。
- `manipulated_sequences/Deepfakes/c23/videos`：DeepFake 伪造视频，标签为 `fake / 1`。

需要完成的工作：

1. 下载 original 和 Deepfakes 视频子集。
2. 使用官方 `train.json`、`val.json`、`test.json` 按视频 ID 划分数据。
3. 每隔 20 帧抽取一张图片，避免连续帧高度重复。
4. 生成 `manifest.csv`，记录图片路径、标签、split、来源视频和抽帧间隔。
5. 检查 real/fake 数量是否平衡，检查坏图和无法解码视频。

已实现脚本：

- [download-FaceForensics.py](../tools/download-FaceForensics.py)
- [extract_ffpp_frames.py](../tools/extract_ffpp_frames.py)

### 3.2 人脸裁剪

训练和推理主路径都使用人脸裁剪图，不直接使用完整视频帧。

需要完成的工作：

1. 使用 OpenCV Haar Cascade 检测人脸位置。
2. 对检测到的人脸框扩大 1.3 倍，保留脸部边缘、下巴、嘴周和融合边界。
3. 选择面积最大的人脸作为主检测目标。
4. 裁剪为正方形并 resize 到 `224x224`。
5. 人脸检测失败时，回退到中心裁剪。
6. 在新的 manifest 中记录 `face_detected`、`bbox`、`crop_path` 等字段。
7. 生成裁剪样本拼图，人工抽查裁剪质量。

### 3.3 频域特征

MVP 阶段使用 FFT 幅度谱作为频域输入，DCT 作为后续改进方向。

需要完成的工作：

```text
face crop RGB
-> grayscale
-> np.fft.fft2
-> np.fft.fftshift
-> log(1 + abs(spectrum))
-> min-max normalize
-> 生成频域特征图
```

频域特征用于捕捉 DeepFake 可能留下的高频伪影、压缩残留和异常纹理分布。

### 3.4 模型训练

模型训练分两步完成：

1. **Baseline 模型**：只使用 RGB 人脸裁剪图进行 real/fake 二分类。
2. **频域-空域融合模型**：RGB 分支学习空域纹理，FFT 分支学习频域伪影，两个分支在特征层 concat 后完成分类。

推荐结构：

```text
RGB face crop -> spatial CNN -> spatial feature
FFT spectrum  -> frequency CNN -> frequency feature

concat(spatial feature, frequency feature)
-> dropout
-> linear classifier
-> fake probability
```

需要完成的工作：

1. 使用 PaddlePaddle 实现 Dataset、DataLoader、模型结构和训练脚本。
2. 支持 baseline 和 fusion 两种模型配置。
3. 使用 CrossEntropyLoss 或二分类 BCE loss。
4. 保存验证集 AUC 最优模型权重。
5. 输出训练日志、验证指标和测试指标。
6. 导出后端推理所需的模型文件和模型版本号。

### 3.5 模型评估

评价指标：

- 主指标：Accuracy、AUC。
- 辅助指标：Precision、Recall、F1。

需要完成的工作：

1. 在 test split 上评估模型。
2. 输出混淆矩阵和指标表。
3. 对比 baseline 与频域-空域融合模型。
4. 保存评估结果到 `reports/` 或 `docs/`，用于答辩和测试报告。

### 3.6 Grad-CAM 可解释性

Grad-CAM 用于展示模型判断 fake 的可疑区域。

需要完成的工作：

1. 对空域分支最后一层卷积特征生成 Grad-CAM。
2. 将热力图 resize 回输入人脸图大小。
3. 叠加到原始图片或人脸裁剪图上。
4. 保存热力图文件路径，供后端返回和前端展示。

## 4. 后端模块

后端使用 Python + FastAPI，负责连接前端、模型和数据库。

### 4.1 主要接口

| 接口 | 方法 | 功能 |
|---|---|---|
| `/api/auth/login` | POST | 用户登录，返回访问令牌或 session |
| `/api/auth/logout` | POST | 用户退出登录 |
| `/api/auth/me` | GET | 获取当前登录用户信息 |
| `/api/detect` | POST | 上传图片并返回检测结果 |
| `/api/records` | GET | 查询当前登录用户的历史检测记录 |
| `/api/records/{id}` | GET | 查询当前登录用户的单条检测详情 |
| `/api/model-version` | GET | 返回当前模型版本 |
| `/api/health` | GET | 服务健康检查 |

除 `/api/health` 和 `/api/auth/login` 外，业务接口需要登录后访问。

历史检测记录按当前登录用户隔离。用户只能查询自己提交的检测记录，不能查看其他用户的图片、热力图或检测结果。

### 4.2 登录认证流程

MVP 阶段采用用户名 + 密码登录，不做公开注册和复杂角色权限。系统可预置一个或多个演示用户，满足课程项目对用户登录、会话保持和访问控制的要求。

```text
用户输入用户名和密码
-> 后端校验用户是否存在
-> 校验 password_hash
-> 登录成功后返回 access token 或写入 session
-> 前端保存登录状态
-> 后续请求携带 token/session
-> 后端校验通过后允许检测和查询历史记录
```

认证模块需要完成的工作：

1. 实现用户名和密码登录。
2. 密码只保存 hash，不保存明文。
3. 登录成功后返回 token 或建立 session。
4. 业务接口校验当前用户身份。
5. 退出登录时清理前端登录状态。
6. 登录失败时返回明确错误信息，但不暴露密码是否正确等敏感细节。

### 4.3 检测接口流程

```text
校验登录状态
-> 接收上传图片
-> 校验文件类型和大小
-> 保存原始图片或计算 hash
-> 调用人脸裁剪
-> 调用 PaddlePaddle 推理
-> 生成 Grad-CAM 热力图
-> 计算 risk level
-> 写入 MySQL 检测记录
-> 返回 JSON 结果
```

返回字段建议：

```json
{
  "record_id": 1,
  "prediction": "fake",
  "fake_probability": 0.86,
  "risk_level": "高风险",
  "original_image_url": "/static/uploads/xxx.jpg",
  "face_crop_url": "/static/crops/xxx.jpg",
  "heatmap_url": "/static/heatmaps/xxx.jpg",
  "created_at": "2026-07-05 12:00:00"
}
```

### 4.4 后端需要完成的工作

1. 搭建 FastAPI 项目结构。
2. 实现登录认证、当前用户获取和退出登录。
3. 实现图片上传和静态文件访问。
4. 封装算法推理服务，避免接口层直接写模型细节。
5. 实现风险等级映射：
   - `< 0.35`：低风险
   - `0.35 - 0.80`：中风险，建议复核
   - `>= 0.80`：高风险
6. 实现检测记录写入和查询。
7. 历史记录查询必须按当前登录用户的 `user_id` 过滤。
8. 增加异常处理：未登录、登录失败、无权访问他人记录、无效图片、无人脸、模型加载失败、数据库连接失败。

## 5. 前端模块

前端使用 Vue + JavaScript，目标是完成一个清晰可演示的检测界面。

### 5.1 页面划分

| 页面 | 功能 |
|---|---|
| 登录页 | 输入用户名和密码，完成登录并进入系统 |
| 图片检测页 | 上传图片、展示原图预览、提交检测 |
| 检测结果页 | 展示原图、人脸裁剪图、伪造概率、风险等级、模型二分类结果、热力图、检测时间和记录编号 |
| 历史记录页 | 查询当前登录用户的历史检测记录，按时间展示原图缩略图、伪造概率、风险等级、模型二分类结果和检测时间 |
| 记录详情页 | 展示单条检测详情、原图、人脸裁剪图、热力图和检测时间 |

### 5.2 前端需要完成的工作

1. 实现登录页面和登录失败提示。
2. 保存登录状态，未登录时跳转登录页。
3. 登录后展示导航入口：图片检测、历史记录、退出登录。
4. 实现图片拖拽上传和文件选择上传。
5. 限制上传格式为 JPG、JPEG、PNG。
6. 调用 `/api/detect` 并处理 loading、成功、失败状态。
7. 使用进度条或环形图展示伪造概率。
8. 用颜色区分风险等级。
9. 展示原图、人脸裁剪图和 Grad-CAM 热力图。
10. 调用 `/api/records` 展示历史记录。
11. 历史记录列表保持简洁，热力图缩略图根据页面空间决定是否展示；如果列表过于拥挤，则只在详情页展示热力图。
12. 保持页面美观、易操作，优先保证演示流程稳定。

## 6. 数据库模块

数据库使用 MySQL，负责保存用户信息、检测记录、模型版本和文件路径。

### 6.1 用户表

建议表名：`users`

| 字段 | 含义 |
|---|---|
| `id` | 主键 |
| `username` | 登录用户名，唯一 |
| `password_hash` | 密码 hash |
| `display_name` | 前端显示名称 |
| `status` | 用户状态，如 active / disabled |
| `created_at` | 创建时间 |
| `last_login_at` | 最近登录时间 |

MVP 阶段不做公开注册，可以通过初始化 SQL 预置演示用户。

### 6.2 检测记录表

建议表名：`detection_records`

| 字段 | 含义 |
|---|---|
| `id` | 主键 |
| `user_id` | 发起检测的用户 ID |
| `image_hash` | 上传图片 hash，用于去重和追溯 |
| `original_image_path` | 原始图片保存路径 |
| `face_crop_path` | 人脸裁剪图路径 |
| `heatmap_path` | Grad-CAM 热力图路径 |
| `prediction` | `real` 或 `fake` |
| `fake_probability` | 伪造概率 |
| `risk_level` | 风险等级 |
| `model_version` | 模型版本 |
| `face_detected` | 是否检测到人脸 |
| `created_at` | 检测时间 |

### 6.3 数据库需要完成的工作

1. 编写 MySQL 建表 SQL。
2. 初始化用户表和演示账号。
3. 初始化数据库连接配置。
4. 提供用户登录查询方法。
5. 提供新增检测记录方法。
6. 提供按用户分页查询历史记录方法。
7. 提供按用户和记录 ID 查询检测详情方法，避免越权访问。
8. 准备演示用测试数据和备份说明。

## 7. 测试模块

测试目标是证明系统主流程能稳定运行。

### 7.1 算法测试

1. 检查抽帧图片数量与 manifest 一致。
2. 检查人脸裁剪输出是否可读。
3. 检查 FFT 特征生成是否无 NaN/Inf。
4. 检查训练脚本能完成至少一个 epoch。
5. 检查推理脚本能输出 fake probability。

### 7.2 后端测试

1. `/api/health` 返回正常。
2. `/api/auth/login` 使用正确用户名密码可登录。
3. `/api/auth/login` 使用错误密码返回失败。
4. 未登录访问 `/api/detect` 和 `/api/records` 时被拒绝。
5. `/api/detect` 登录后能接收合法图片。
6. 非图片文件上传时返回错误。
7. 检测完成后数据库新增记录。
8. `/api/records` 能查询当前用户历史记录。
9. 用户不能通过记录 ID 查询其他用户的检测详情。

### 7.3 前端测试

1. 登录页面输入和错误提示正常。
2. 未登录时不能进入检测和历史记录页面。
3. 图片上传预览正常。
4. 检测 loading 状态正常。
5. 结果、概率、风险等级和热力图展示正常。
6. 历史记录页面能正常加载。

## 8. 集成流程

推荐按下面顺序集成：

```text
算法先产出推理脚本和模型文件
-> 数据库完成用户表和检测记录表
-> 后端完成登录认证
-> 后端封装模型推理服务
-> 后端完成上传和记录接口
-> 前端接入登录和检测接口
-> 数据库保存检测记录
-> 联调热力图静态访问
-> 完成功能测试和演示数据准备
```

每个模块交付时必须明确输入和输出：

- 算法给后端：模型文件、推理函数、输入尺寸、标签含义、模型版本。
- 后端给前端：接口地址、请求格式、响应 JSON、错误码。
- 数据库给后端：表结构、连接配置、查询方法。
- 前端给全组：可演示页面和操作流程。
